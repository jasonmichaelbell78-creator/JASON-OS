## Learnings for Methodology

### 1. session-bookmark Pattern Has Three Variants
The RESUME.md / session-bookmark pattern appears in three structural forms:
1. **Standalone session dir** (`session-NNN/RESUME.md`) — pure bookmark, no plan content
2. **Feature-embedded RESUME** (`feature-dir/RESUME.md`) — mid-feature handoff alongside PLAN.md
3. **Wave-scoped resume** (`feature-dir/WAVE4_RESUME.md`) — execution-phase-specific, self-describes as deletable

These should be distinguished in the schema. Currently `execution-handoff` and `session-bookmark` overlap. Suggestion: add `variant` field or use sub-types.

### 2. execution-handoff Has Multiple Sub-Types
- `RESUME.md` = general session context handoff
- `HANDOFF.md` = deferred phase execution handoff (inputs + deliverables for a specific phase)
- `STEP_A_HANDOFF.md` = mid-step handoff (specific plan step context)
- `REMAINING_TASKS.md` = punch-list handoff (enumerated remaining work items)
- `pr-body-session-N.md` = PR body draft to survive context clear

The JSONL schema's `plan_scope: execution-handoff` conflates these. D22 may want to add a `handoff_sub_type` field.

### 3. SWS JSONL Registry Pattern is Highly Portable
The `system-wide-standardization/` dir uses machine-readable JSONL registries (decisions, directives, ideas, tenets) alongside narrative planning docs. This is the most schema-mature pattern in the repo and a strong candidate for the JASON-OS planning standard.

### 4. No PORT_ANALYSIS.md in SoNash
Piece 1a §1 identified PORT_ANALYSIS.md as a key target. It does not exist in SoNash's `.planning/`. The portability analysis happens in the JASON-OS research artifacts (this scan), not in SoNash's own planning.

### 5. STATE.md Was the Original Session-Bookmark Predecessor
`archive/STATE.md` (v3.2) was a project-wide state tracker that predates per-session RESUME.md files. The evolution: STATE.md (shared, versioned) → per-session RESUME.md (scoped, disposable). This transition happened organically during Sessions #237-#245.

### 6. Planning Dir Sizes Are Highly Variable
Range: 5KB (t28-intelligence-graph-v1) to 598KB (system-wide-standardization). Feature plans cluster around 25-95KB. The SWS outlier suggests machine-readable JSONL can inflate planning dirs significantly.

### 7. plan-orchestration/CL-PROTOCOL.md is a Cross-Plan Methodology Doc
This document is portable (no SoNash-specific content) and represents a cross-PR learning. It lives in `.planning/plan-orchestration/` but its scope is the entire planning system. A future JASON-OS port should elevate it to `.claude/skills/_shared/` or equivalent.

### 8. Agent Sizing Was Appropriate
D15b scope was well-sized: 17 active dirs + 4 top-level files + archive was manageable in a single pass. D15a's absence (research-program dirs) was gap-filled without scope overflow.

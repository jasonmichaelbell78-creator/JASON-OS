## Pre-flight observations (before any agents spawn)

### [ORCH-BUG] PLAN.md §S10 invocation was fictional

**Symptom:** PLAN.md §S10 described the back-fill as invokable via
`node .claude/sync/label/backfill/orchestrate.js --repo=jason-os`, but
`orchestrate.js` has no CLI entry point. Its own docstring says "Agent
dispatch is INJECTED — the module itself never spawns Claude Task tool
agents."

**Example:** A future contributor (human or Claude) following the plan
verbatim would try to run the node command, hit "nothing happens" (the
module just exports functions), and have to reverse-engineer the actual
Claude-driven flow.

**Impact:** MEDIUM (documentation / onboarding). Back-fill was runnable
once someone realized it; the time cost was reading ~300 lines to
discover the contract.

**Recommendation:** §S10 patched in this run to describe the
Claude-driven flow truthfully. For Piece 5 `/sync`, decide up front
whether flows that require Claude-session drivers are invoked via
skills (live Claude) or CLI (standalone) and document that explicitly
in the SKILL.md / CLI help.

### [SCOPE-REFINE] 17 empty `.husky/_/*` shims all in scope

**Symptom:** scope.json `.husky/**/*` include pattern sweeps in 17
husky-generated hook stubs that are empty (0.0 KB). They'll each get
a near-identical catalog record with `status: generated` /
`portability: generated`.

**Example:** `.husky/_/applypatch-msg`, `.husky/_/commit-msg`,
`.husky/_/post-*`, `.husky/_/pre-*`, etc. All generated by husky's
`install` command.

**Impact:** LOW (catalog bloat). 17 near-identical records is
pure noise; none of them will ever be asked about individually.

**Recommendation:** Consider adding `.husky/_/**` to scope.json
excludes (or narrowing husky include to the user-authored hooks like
`.husky/pre-commit` + `.husky/pre-push` + `.husky/_shared.sh`).
Defer decision until after the run — if the records end up useful
for `/sync` to reproduce the husky install in target repos, keep.
Otherwise trim.

### [SCOPE-REFINE] Single 96 KB file is a batch of one

**Symptom:** `.planning/PR_REVIEW_LEARNINGS.md` at 96.5 KB is twice
the size of the next-largest file and gets B01 to itself. An agent
deriving this file must read the entire 96 KB before emitting a
single record.

**Example:** B01 = [PR_REVIEW_LEARNINGS.md alone].

**Impact:** LOW-MEDIUM (agent time, potential shallow read). Agent
may skim rather than fully internalize content, producing a shallower
`purpose` / `dependencies` derivation.

**Recommendation:** After this run, if B01's record looks shallow,
consider:
- Adding a "read file head + structure only" hint to agent prompts
  for files >50 KB (since `purpose` and `type` rarely depend on
  body content for long narrative docs)
- OR: flag `purpose_confidence: low` on oversized-file records and
  let `/label-audit` re-derive later with a different strategy

## Per-batch observations

_Populated as batches run. One subsection per batch with any issues observed._

### B01 — `.planning/PR_REVIEW_LEARNINGS.md` (96.5 KB, 1 file)

**Outcome:** Both agents produced semantically-valid records with
meaningful agreement. Cross-check flagged 5 disagreements (purpose wording,
dependencies, notes wording, data_contracts, content_hash).

**Findings:**

### [SCHEMA-GAP] `confidence` field not in file_record schema

**Symptom:** `schema-v1.json` defines `file_record` with
`additionalProperties: false` but does not list `confidence` as a property.
Yet the agent template instructs every agent to emit a top-level
`confidence` object (one key per field, values in [0.0, 1.0]). Running raw
agent output through ajv produces `(root): must NOT have additional properties`.

**Example:** B01 primary + secondary both rejected by ajv at the
`confidence` field until stripped.

**Impact:** MEDIUM. Any validator that receives raw agent output without
stripping confidence first will falsely reject valid records. Cross-check
code already strips it (see `cross-check.js`); the gap is that
`verify.js` had to replicate the strip, and any future tooling must too.

**Recommendation:** Add `confidence` to the `file_record` schema as a
permitted object (with `additionalProperties: {type: "number"}`), marked
explicitly as "runtime metadata — not persisted to catalog after
cross-check." This is self-documenting and removes the
strip-before-validate dance.

### [SCHEMA-GAP] `content_hash: null` rejected by schema

**Symptom:** Schema defines `content_hash` as `{type: "string"}` — no
nullable alternative. Agent-primary-template says "SHA of content at scan
time (agent-runner provides this)" — implying agents should not emit it —
but the dispatch wrapper (mine) said "otherwise null" which produced
`content_hash: null` on primary, rejected by schema.

**Example:** B01 primary `/content_hash: must be string`.

**Impact:** MEDIUM. Agents that emit `null` here get rejected even
though the intent (unknown, runner will fill) is perfectly valid.

**Recommendation:** Schema should make `content_hash` a union type:
`{oneOf: [{type: "string"}, {type: "null"}]}`. OR the template should
explicitly say "OMIT this field if you cannot compute it — do not emit
null." I'll fix the dispatch wrapper for remaining batches to say "omit
if unable."

### [PROMPT-DRIFT] Dependencies semantics — "writer" vs "dependency"

**Symptom:** B01 primary treated `/pr-review` skill as a HARD dependency
of `PR_REVIEW_LEARNINGS.md` ("pr-review writes to me"). B01 secondary
left `dependencies: []` (file is the OUTPUT of the skill, not a consumer
of it). Both are defensible; they represent different semantic
interpretations of "dependency."

**Example:** B01 `dependencies` disagreement.

**Impact:** MEDIUM. If this interpretation split recurs across batches,
the catalog becomes inconsistent — some output-files list their writers,
others don't. Downstream `/sync` planning (who needs what ported
together) becomes unreliable.

**Recommendation:** CATALOG_SHAPE.md §3.3 needs an explicit rule: an
output-only file does NOT list its writer as a dependency. The
writer/producer relationship belongs in `data_contracts` (with
`role: "consumer"` on the log file's side is arguably wrong — the log is
a `producer` output, the skill is the `consumer` writer). Frame it: "A
depends on B" means "if B is absent, A can't do its job" — a passive log
file still exists without its writer.

### [PROMPT-DRIFT] `plan_scope` ambiguity — `cross-pr-learning` vs `learnings`

**Symptom:** Both agents correctly converged on `plan_scope:
"cross-pr-learning"` but both flagged it with confidence < 0.85 because
`learnings` is also a valid enum value.

**Impact:** LOW. They got it right; the confidence flags are honest.

**Recommendation:** SCHEMA.md §8.18 could add one-line disambiguation
guidance per enum value: "cross-pr-learning = spans multiple PRs;
learnings = single PR post-mortem or single research post-mortem."

**Throughput:** Primary 63s, Secondary 62s (parallel wallclock ~63s).
Each agent used ~62-64k tokens and 11-12 tool calls.

### B02 — schema + docs, 11 files (husky shim-heavy)

**Outcome:** Heavy findings. Both agents returned. Primary cleaned
post-hoc to fix MY prompt drift ("generated" instruction). Secondary
was cleaner initially but hit a template-vs-schema contract break on
`lineage` that tanked 8/11 records.

### [SCHEMA-GAP] **CRITICAL: lineage shape in template contradicts schema**

**Symptom:** `agent-primary-template.md` §3.4 says lineage =
`{source_repo, source_path, source_version, notes}`.
`schema-v1.json` `lineage_object` requires
`{source_project, source_path, source_version, ported_date}` with
`additionalProperties: false`. `source_repo` and `notes` are rejected
by the schema; `source_project` and `ported_date` are required.

**Example:** B02 secondary emitted lineage objects for 8 husky files
using the template shape. ALL 8 failed schema validation:
- `must NOT have additional property 'source_repo'`
- `must have required property 'source_project'`
- `must have required property 'ported_date'`

**Impact:** HIGH. Any lineage-bearing record following the template
verbatim gets rejected by ajv. If the catalog ships with broken
lineage records, validate-catalog.js pre-commit blocks every future
commit involving those files.

**Recommendation:** IMMEDIATE — fix `agent-primary-template.md` and
`agent-secondary-template.md` to match the schema. Add a proper
Lineage Records section to SCHEMA.md (currently §3 table points at
"§4" for lineage but §4 is Section Records — a doc bug).

### [SCHEMA-GAP] `required_secrets` / `tool_deps` / `external_services`: schema wants objects, template implies strings

**Symptom:** Template says "required_secrets — env var names
(SONAR_TOKEN, GITHUB_TOKEN)" — implying bare strings.
`schema-v1.json` defines these as arrays of `external_dep_entry` =
`{name, hardness}` objects.

**Example:** B03 primary emitted
`required_secrets: [{name: "GITHUB_TOKEN", hardness: "hard"}]`
(correct per schema). verify.js SANITY layer incorrectly assumed
strings.

**Impact:** MEDIUM. Agents drift to string form because template
examples look like bare strings. Fixed verify.js to accept both shapes.

**Recommendation:** Rewrite template §3.3 with explicit JSON examples
for `required_secrets`, `tool_deps`, `external_services`,
`mcp_dependencies` — all `external_dep_entry` per schema.

### [PROMPT-DRIFT] MY dispatch wrapper told agents to use "generated" for portability

**Symptom:** B02 dispatch prompt instructed "label as `status:
generated`, `portability: generated`". But `portability` enum is
only `portable | sanitize-then-portable | portable-with-deps |
not-portable | not-portable-product`. "generated" is valid for
status, not portability.

**Example:** B02 primary emitted `portability: "generated"` verbatim
on 8 husky files (confidence 0.0 because the agent caught it).
Secondary chose `not-portable` independently. I post-hoc corrected
primary to match secondary.

**Impact:** HIGH this batch. Contaminated primary output with
instruction-induced bad values.

**Recommendation:** Dispatch wrapper for B04+ drops this
instruction; explicitly lists the 5 valid `portability` values.

### [PROMPT-DRIFT / SCHEMA-GAP] `type: hook` misapplied to git hooks

**Symptom:** §9.3 `hook`/`hook-lib` extension is for Claude Code
events (§8.5 enum: PreToolUse, PostToolUse, etc.) — NOT git hooks.
Git hooks (applypatch-msg, commit-msg) have path `.husky/_/...`
which looks hook-like.

**Example:** B02 primary originally used `type: "hook"` with
`event: "PreToolUse"` (confidence 0.0). B02 secondary used
`type: "other"`.

**Impact:** MEDIUM.

**Recommendation:** Either (a) add `git-hook` to §8.1 type enum + §9
extension, or (b) explicitly document in template that git-hook
files use `type: "other"` until a git-hook type lands. I'm using (b)
for the remaining batches and filing (a) as a todo.

### [DERIVE-MECH] detectType returns "other" too often

**Symptom:** `detectType()` returned `"other"` for `.validate-test.cjs`
(a test script) and `husky.sh` (a shell script). Agents correctly
chose `script` / `script-lib` and flagged the disagreement.

**Impact:** LOW-MEDIUM. Every reasonable agent judgment creates a
noise warning.

**Recommendation:** Extend `detectType()` rules: `.cjs|.mjs` → script
or script-lib; `.sh` → script. File as a todo before `/sync` work.

**Throughput (B02):** Primary 129s, Secondary 120s. 65-72k tokens
each, 17 tool calls each.

### B03 — skill + doc + ci-workflow, 3 files

**Outcome:** Both agents schema-clean post-verify. 17 disagreements
— mostly word-choice. Independent convergence on correct types:
- `REFERENCE.md` → `type: "doc"` (both fell back correctly when
  `skill-reference` wasn't in the enum)
- `skill-audit/SKILL.md` → `type: "skill"` with §9.1 extensions
- `auto-merge-dependabot.yml` → `type: "ci-workflow"` with §9.9

### [PROMPT-DRIFT] SKILL.md refs files that don't exist in JASON-OS

**Symptom:** B03 primary flagged that `skill-audit/SKILL.md`
references `scripts/skills/skill-audit/self-audit.js` and
`.claude/skills/_shared/SELF_AUDIT_PATTERN.md` — neither exist in
JASON-OS. Primary listed both as `hard` deps; secondary did not.

**Impact:** MEDIUM-HIGH as a pattern. Dangling `hard` deps in the
catalog would fail `crossBatchConsistency` at end of run.

**Recommendation:** Add to dispatch prompt: "Before listing a file
as `hard` dep, verify it exists. Use `soft` otherwise + note in
`notes`." Also filed as todo: `skill-audit/SKILL.md` needs cleanup
(references missing infrastructure).

### [ORCH] Primary vs Secondary dep counts diverged

**Symptom:** `skill-audit` — primary 6 deps, secondary 3. Not
"who's wrong" — "who's more thorough." Primary superset caught
dangling refs; secondary quietly elided them.

**Impact:** MEDIUM.

**Recommendation:** Synthesis rule: keep primary's superset;
downgrade `hard → soft` on entries not found on disk; note as
missing.

**Throughput (B03):** Primary 117s, Secondary 123s. 64-74k tokens
each, 18-24 tool calls each.

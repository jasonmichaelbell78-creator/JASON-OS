## Learnings for Methodology

### Agent sizing

D8 was right-sized for this scope. 13 catalogued units across 5 top-level entries and 2 subdirs (15 files total). Reading all top-level files (3 files) + key files per subdir (PLAN.md, PORT_ANALYSIS.md header, BOOTSTRAP_DEFERRED.md, HANDOFF.md, RESUME.md) + sampling DECISIONS.md was achievable in a single agent pass. If subdirs had been deeper (like SoNash's 21 dirs with 394 files), this agent would need splitting by subdir or by file type (plan files vs support files vs learnings docs).

### File-type observations

JASON-OS planning trees have a consistent pattern per planning session:
- One PLAN.md (the implementation plan — always the largest or second largest)
- One DECISIONS.md (decisions log — always present)
- Supporting docs vary: DIAGNOSIS.md/HANDOFF.md/RESUME.md in implementation sessions; DEPENDENCIES.md/SYNTHESIS.md/RESEARCH_ROADMAP.md in research sessions
- PORT_ANALYSIS.md appears only in implementation sessions with SoNash port steps

The todos.jsonl + TODOS.md pairing (machine-readable + human-readable) is a noteworthy pattern: they are not planning session artifacts but cross-session backlog infrastructure living alongside planning dirs. SoNash will likely have a similar or richer version.

### Classification heuristics

Planning artifacts are consistently `project + not-portable` with one exception: learnings docs (PR_REVIEW_LEARNINGS.md) contain transferable methodology learnings worth `sanitize-then-portable`. The exception rule: if a file's primary content is process methodology (how we reviewed, what patterns we saw, what rules we derived) rather than project state (what we built, what decisions we made for this project), flag as `sanitize-then-portable`. The BOOTSTRAP_DEFERRED.md pattern (deferral registry with revisit triggers) is also partially transferable as a methodology pattern, but its content is fully project-specific.

### Dependency extraction

Planning files reference research dirs prominently and explicitly:
- PLAN.md header: "References: `.research/jason-os-mvp/RESEARCH_OUTPUT.md`"
- PLAN.md body: references to specific finding files (G2, G4, G5, G6) in `.research/jason-os-mvp/findings/`
- DECISIONS.md: references decisions grounded in research (D8, D22 cite research outputs)

These links are prose references (not structured metadata) — easy to extract via reading but not machine-parseable without NLP. For SoNash's 394 files, an agent would need to read each PLAN.md header + first section to capture research dependencies. Consider making this a schema field: `research_inputs: [<research dir slug>]`.

### Schema-field candidates

Planning sessions have attributes worth standardizing as schema fields:
- `plan_scope`: milestone | research-program | deferral-registry | session-bookmark | port-ledger | execution-handoff
- `session_id`: maps to deep-plan invocation (e.g., "jason-os-mvp")
- `status`: active | complete | deferred | gated
- `layer_coverage`: list of layer IDs covered (e.g., ["0+", "0", "1", "step-4"])
- `port_count`: number of rows in PORT_ANALYSIS.md (for implementation sessions)
- `decision_count`: number of decisions in DECISIONS.md
- `research_inputs`: list of research dir slugs consumed
- `next_action`: free-text or structured pointer to what comes next
- `pr_count`: number of PRs produced

RESUME.md-style files are a distinct artifact type worth a `session-bookmark` plan_scope variant. They carry git state (SHA, branch), what-is-done summary, what-is-next pointer, and outstanding user actions — reusable inter-session communication pattern.

### Adjustments recommended for SoNash planning scan

SoNash has 21 planning dirs with 394 files. Recommended adjustments:

1. **Split by session type, not alphabetically.** Group agents: (a) research-program sessions, (b) implementation/port sessions, (c) feature-specific sessions. Each type has different key files (PLAN.md + DECISIONS.md + SYNTHESIS + roadmap vs PLAN.md + PORT_ANALYSIS + DIAGNOSIS + HANDOFF).

2. **PORT_ANALYSIS.md files deserve dedicated agents.** In JASON-OS, PORT_ANALYSIS.md is 53KB with 33+ rows. SoNash likely has larger port ledgers. Assign one agent per PORT_ANALYSIS.md to extract row-level stats (verdict distribution, total port count, blocked items).

3. **Sample DECISIONS.md, do not read fully.** DECISIONS.md at 23KB contains 38 decisions. For census purposes, read the first 2 sections to capture decision_count, scope, and meta-instruction count — then stop. Full read only if a specific decision is needed.

4. **RESUME.md and HANDOFF.md are quick reads — always read them.** They are session bookmarks (4-5KB each) that capture the current state more precisely than reading an entire PLAN.md. Start with these for status extraction.

5. **Plan dependency links require reading PLAN.md headers.** Research inputs are embedded in plan prose. For 21 dirs, this adds ~21 file reads but yields the research-to-plan linkage graph that Piece 2 schema design needs.

6. **Learnings docs need explicit tagging in the schema.** PR_REVIEW_LEARNINGS.md sits at `.planning/` root (not in a subdir). SoNash may have equivalent learnings docs in various locations. Flag them as `learnings-doc` type regardless of location.

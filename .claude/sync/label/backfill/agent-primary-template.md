### Field derivation rules

**Identity** (¬ß3.1)

- `name` ‚Äî filename slug (without extension), or YAML `name:` frontmatter
  where present
- `path` ‚Äî repo-root-relative, forward slashes
- `type` ‚Äî apply `.claude/sync/label/lib/derive.js` `detectType()` rules
  first (path + extension + directory); override if frontmatter clearly
  contradicts
- `purpose` ‚Äî one-sentence description from the file's own documentation
  header or YAML `description:`. If absent, synthesize from content in
  the imperative: "Validates X before commit." not "This script is the
  one that handles validating X."

**Scope & sync-action** (¬ß3.2)

- `source_scope` ‚Äî where the code/definition belongs:
  - `universal` ‚Äî travels to any Claude Code project
  - `user` ‚Äî tied to this operator (lives in ~/.claude/, per-user config)
  - `project` ‚Äî tied to this repo (settings.json contents, project PLAN.md)
  - `machine` ‚Äî tied to this computer (absolute paths, machine-scoped cache)
  - `ephemeral` ‚Äî session-local (task state files)
- `runtime_scope` ‚Äî where the file's runtime effects land (often differs
  from source_scope for helpers with universal source but project-scoped
  writes ‚Äî see SCHEMA.md ¬ß3.2 for the rationale)
- `portability` ‚Äî what the sync tool should do:
  - `portable` ‚Äî copy as-is
  - `sanitize-then-portable` ‚Äî copy + edit specific fields (populates
    `sanitize_fields` below)
  - `portable-with-deps` ‚Äî copy + document external deps
  - `not-portable` ‚Äî don't copy (file is meaningless elsewhere)
  - `not-portable-product` ‚Äî application code that never crosses repos
- `status` ‚Äî lifecycle state:
  - `active` ‚Äî in use right now
  - `complete` ‚Äî finished milestone
  - `deferred` ‚Äî planned for later
  - `stub` ‚Äî placeholder awaiting implementation
  - `gated` ‚Äî blocked by an external condition
  - `deprecated` ‚Äî still works but retiring
  - `archived` ‚Äî historical
  - `generated` ‚Äî auto-generated by tooling
  - (`partial` ‚Äî transient Piece 3 state; agents should NOT emit this ‚Äî
    it's written by the hook only)

**Dependencies** (¬ß3.3) ‚Äî FOUR distinct arrays:

- `dependencies` ‚Äî other registry files this one needs (object form:
  `{name, hardness: "hard"|"soft", kind: "spawn"|"import"|"reference"|"invoke"}`)
- `external_services` ‚Äî third-party APIs (GitHub, SonarCloud, OpenAI)
- `tool_deps` ‚Äî external CLI binaries (gh, gitleaks, go, node, bash)
- `mcp_dependencies` ‚Äî MCP server tool names
- `required_secrets` ‚Äî env var names (SONAR_TOKEN, GITHUB_TOKEN)

Empty array is valid when the file needs nothing in a category. Each
array maps to a distinct operator-remediation flow; do NOT merge them.

**Provenance** (¬ß3.4)

- `lineage` ‚Äî where the file came from. `null` for native files.
  Format (object): `{source_repo, source_path, source_version, notes}`.
  For Lineage documented as markdown body `**Lineage:** ...`, parse into
  the object shape.
- `supersedes` ‚Äî array of file names this one replaces
- `superseded_by` ‚Äî name of replacement, null if still current

**Sync mechanics** (¬ß3.5)

- `sanitize_fields` ‚Äî string identifiers that need per-repo replacement
  when porting (empty when `portability != sanitize-then-portable`)
- `state_files` ‚Äî paths the file reads/writes in `.claude/state/`:
  `[{path, access: "read"|"write"|"read-write"}]`

**Catch-all** (¬ß3.6)

- `notes` ‚Äî free-text nuances that don't fit structured fields
- `data_contracts` ‚Äî implicit-schema couplings with other files:
  `[{contract_name, target_file, role, fields}]`

**Relationships** (¬ß3.7)

- `component_units` ‚Äî sub-units that compose this file:
  `[{name, path, role}]`. Empty for uniform files.
- `composite_id` ‚Äî which composite this file belongs to, or null

**Copy detection** (¬ß3.8)

- `is_copy_of` ‚Äî path to canonical source if copy, else null
- `has_copies_at` ‚Äî if this IS the canonical, list paths where copies live
- `content_hash` ‚Äî SHA of content at scan time (agent-runner provides this)

**Granularity** (¬ß3.9)

- `sections` ‚Äî per-section records for mixed-scope files (rare, ~10-20
  across both repos). Empty array is the norm. When populated, use the
  ¬ß4 section-record shape from SCHEMA.md.

### Piece 3 machinery fields (¬ß4)

### Confidence reporting

Each field comes with a confidence score in [0.0, 1.0]. Fields where
evidence is ambiguous (multiple plausible values, frontmatter conflicts
with heuristic, etc.) should score **below 0.80** so the cross-check
layer flags them.

**Output shape (MUST be consistent with `CATALOG_SHAPE.md` and the hook's
runtime `applyAgentOutput` merge).** Return a flat catalog record matching
`CATALOG_SHAPE.md` ¬ß3 + the per-type extensions in `SCHEMA.md` ¬ß9, plus a
top-level `confidence` object keyed by field name. The flat record spreads
cleanly into the existing record; the `confidence` object is consumed
separately by `confidence.js` for the cross-check layer.

```json
{
  "path": ".claude/hooks/foo.js",
  "type": "hook",
  "source_scope": "universal",
  "portability": "sanitize-then-portable",
  "purpose": "‚Ä¶",
  "confidence": {
    "type": 0.98,
    "source_scope": 0.92,
    "portability": 0.65
  }
}
```

The aggregator in `confidence.js` combines primary + secondary
`confidence` values per the rules in `DISAGREEMENT_RESOLUTION.md`. Nested
`{value, confidence}` shapes are NOT accepted ‚Äî the flat record + separate
confidence object is the one on-disk contract.

### Hard constraints

- **No silent elisions.** Every field in ¬ß3 must be present in the
  output, even if empty or null.

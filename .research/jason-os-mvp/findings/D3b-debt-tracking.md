## Key Findings

### 1. `/add-debt` Skill ŌĆö How Debt Gets Logged [CONFIDENCE: HIGH]

SoNash has a fully-specified `add-debt` skill at
`.claude/skills/add-debt/SKILL.md` (v2.0, 2026-03-20). It is the single
interactive entry point for logging individual debt items and operates in two
modes:

- **Deferred mode** (PR number provided): produces source IDs like
  `PR-{number}-{seq}`. Used during `/pr-review` to log items deferred from
  review. S0 items cannot be deferred ŌĆö they must be fixed before PR merges or
  downgraded.
- **Manual/ad-hoc mode** (no PR number): source ID is `manual`. Used whenever
  debt is discovered outside formal audits.

The skill collects: `file`, `line`, `title`, `severity` (S0-S3), `category`
(9 values: security, performance, code-quality, documentation, refactoring,
process, engineering-productivity, enhancements, ai-optimization), `type`
(bug|code-smell|vulnerability|hotspot|tech-debt|process-gap|enhancement),
`effort` (E0-E3), `description`, and optionally `pr_number` + `reason` for
deferrals.

Execution dispatches to one of two Node.js scripts:
- `node scripts/debt/intake-manual.js` ŌĆö for manual items
- `node scripts/debt/intake-pr-deferred.js` ŌĆö for PR-deferred items

Both scripts run SHA256 content-hash deduplication (on normalized
`file + line + title + description`) and write atomically to MASTER_DEBT.jsonl
via a shared `appendMasterDebtSync` function. After intake the skill calls
`node scripts/debt/generate-views.js` to regenerate markdown views.

The skill does NOT handle: bulk imports from audits (use `intake-audit.js`
directly), automated hook/CI sources (same), or items already in
MASTER_DEBT.jsonl. [Source: `sonash-v0/.claude/skills/add-debt/SKILL.md`]

### 2. Debt Storage ŌĆö MASTER_DEBT.jsonl as Canonical Store [CONFIDENCE: HIGH]

Primary storage: `docs/technical-debt/MASTER_DEBT.jsonl` ŌĆö one JSON object per
line, 8,505 items as of the time of this research. Each record has a stable
`DEBT-XXXX` sequential ID plus a `content_hash` for dedup.

Full field schema (from inspecting actual records):
```
id, source_id, source_file, category, severity, type, file, line, title,
description, recommendation, effort, status, roadmap_ref, created,
verified_by, resolution, content_hash, original_id, merged_from, source,
rule, sonar_key
```

Status values observed in the data: `NEW`, `VERIFIED`, `IN_PROGRESS`,
`RESOLVED`, `FALSE_POSITIVE`. Note: `TRIAGED` exists in research-identified
schema proposals but is absent from current MASTER_DEBT records.

### 3. Debt Lifecycle ŌĆö Classification, Priority, Resolution [CONFIDENCE: HIGH]

Items move through a defined lifecycle with `status` as the state field:

```
NEW (intake) -> VERIFIED (convergence-loop verify mode) -> IN_PROGRESS
             -> RESOLVED (via resolve scripts or CI workflow)
             -> FALSE_POSITIVE (manual clearing or dedup)
```

**Severity** drives prioritization (S0=critical, S1=high, S2=medium, S3=low).
S0 security items trigger a recommendation to route to `/security-auditor`
before resolution. Effort estimates (E0=<30m, E1=<2h, E2=<8h, E3=>8h) inform
plan ordering.

**Resolution** happens through:
1. Manual: `node scripts/debt/resolve-item.js DEBT-XXXX --pr <PR#>`
2. Bulk resolve: `node scripts/debt/resolve-bulk.js --file staging/corrections.jsonl`
3. Automated: `.github/workflows/resolve-debt.yml` ŌĆö triggers on PR merge.
   Parses PR body for `Resolves: DEBT-XXXX` lines (strict format: only
   `Resolves:` prefix lines, not mentions elsewhere). Calls `resolve-bulk.js
   --eligible-only` then commits changes to `docs/technical-debt/` and pushes.
   Items in `NEW` status are ineligible for auto-resolution (must be VERIFIED
   or IN_PROGRESS).

**Verification** happens in `debt-runner verify` mode ŌĆö spawns convergence-loop
agents to check whether each item's file still exists and the issue is still
present. Corrections are staged, reviewed, then applied.

**Dedup** is multi-pass via `dedup-multi-pass.js` ŌĆö content-hash similarity
>80% triggers merge candidates. S0/S1 items cannot be silently absorbed into
lower-severity merges (guard rail in debt-runner).

[Sources: `sonash-v0/.claude/skills/debt-runner/SKILL.md`;
`sonash-v0/.github/workflows/resolve-debt.yml`;
`sonash-v0/.claude/skills/debt-runner/REFERENCE.md` staging schemas]

## Contradictions

**1. `hasDebtCandidates` location in metadata.json: top-level vs consumerHints**

REFERENCE.md Section 11 schema template places `hasDebtCandidates` only inside
`consumerHints`. Actual metadata.json files (t28, multi-layer-memory,
github-health) place it both at top-level AND inside `consumerHints`. The
SKILL.md Phase 5 routing reads `metadata.json hasDebtCandidates: true` without
specifying the JSON path ŌĆö ambiguous whether it reads top-level or
`consumerHints.*`. The actual synthesizer agent would need to be inspected to
resolve definitively. For JASON-OS porting purposes, writing both (top-level +
consumerHints) is the safer choice.

**2. debt-runner SKILL.md says 7 modes; REFERENCE.md Section 11 example metadata
format shows no debt-runner integration**

The SKILL.md is authoritative. The 7-mode structure is confirmed. REFERENCE.md
does not define a debt-runner-specific schema extension ŌĆö it is already embedded
in the SKILL.md.

## Serendipity

**`escalate-deferred.js` is invisible.** The script `scripts/debt/escalate-deferred.js`
auto-escalates items deferred multiple times but appears in no skill
documentation (confirmed by the SQ7a research finding). The intake-log.jsonl
records escalation events, but no skill surfaces them. This is an invisible
pipeline step that JASON-OS can simply omit from v0 without any user-visible
loss.

**`known-debt-baseline.json` is a ratchet pattern, not just debt storage.**
It stores baselines for quality thresholds (raw-error-message count, cyclomatic
complexity per file). This is a separate concept from MASTER_DEBT.jsonl ŌĆö it
enforces "no regression from N" rather than tracking individual items. JASON-OS
could adopt this pattern independently of the full TDMS stack as a lightweight
quality gate.

**The `PromoteToDebtButton` pattern** (documented in dev-dashboard research) is
a generalizable UI primitive ŌĆö any tool or skill that encounters a
track-as-debt-worthy event can generate a clipboard-copied `/add-debt` invocation
string. This is a zero-infrastructure-cost pattern that JASON-OS skills could
adopt before a full TDMS is in place.

**The dead-end is low-severity.** The `hasDebtCandidates` routing in
`deep-research` is menu item 8 (the last item, after the main content menu).
If the user triggers it and `/add-debt` doesn't exist, Claude Code will simply
error on the skill invocation. It does not corrupt any research output or block
any primary workflow. The dead-end is annoying but not catastrophic ŌĆö JASON-OS
can ship deep-research without add-debt and add it later without rework.
